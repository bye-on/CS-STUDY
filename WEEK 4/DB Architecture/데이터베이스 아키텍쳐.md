# 데이터베이스 아키텍쳐

# 모놀리식 아키텍처

모놀리식 아키텍처는 애플리케이션의 모든 구성 요소(UI, 비즈니스 로직, 데이터 접근 계층 등)가 하나의 코드 베이스로 통합되어 단일 실행 파일로 배포되는 방식이다.

| **구분** | **장점 (Pros)** | **단점 (Cons)** |
| --- | --- | --- |
| **개발** | 초기 설정이 간단하고 개발이 빠름 | 기능이 많아질수록 코드 이해 및 수정 어려움 (높은 결합도) |
| **테스트** | 단일 통합 테스트가 용이함 | 전체 시스템을 테스트해야 하므로 시간이 오래 걸릴 수 있음 |
| **배포** | 한 번에 전체를 배포하여 간단함 | 작은 변경에도 전체 애플리케이션을 다시 배포해야 함 |
| **운영/확장** | 관리할 요소가 적어 운영이 단순함 | 특정 기능에 부하가 몰려도 전체 시스템을 확장해야 함 (확장성 제한) |
- **소규모 애플리케이션 또는 스타트업 초기 단계:** 복잡도가 낮고 빠르게 시장에 출시해야 할 때.
- **팀 규모가 작을 때:** 여러 팀 간의 협업 및 조정이 필요 없을 때.
- **기술적 요구사항이 단순하고 안정적일 때:** 장기적인 확장이나 유연성보다는 단순한 구조가 중요할 때.

# 마이크로서비스 아키텍처(MSA)

MSA는 애플리케이션을 독립적으로 배포 가능한 작은 서비스들의 집합으로 구성하는 방식이다. 각 서비스는 자체 비즈니스 로직과 데이터베이스를 가질 수 있으며 API를 통해 통신한다.

| **구분** | **장점 (Pros)** | **단점 (Cons)** |
| --- | --- | --- |
| **개발** | 서비스별 독립적 개발, 빠른 배포 사이클, 기술 스택 유연성 | 복잡한 서비스 간 통신 관리, 분산 트랜잭션 처리의 어려움 |
| **테스트** | 서비스 단위의 독립적 테스트가 가능함 | 전체 시스템의 통합 테스트 및 모니터링이 복잡함 |
| **배포** | 서비스별 독립적 배포 및 롤백이 가능함 | 관리해야 할 배포 단위와 인프라가 많아짐 (DevOps 복잡도 증가) |
| **운영/확장** | 서비스별 독립적 확장 (수평 확장 용이), 내결함성이 높음 | 운영 및 모니터링 복잡성 증가, 높은 네트워크 오버헤드 |
- **대규모 및 복잡한 엔터프라이즈 애플리케이션:** 시스템이 너무 커져서 모놀리식 구조의 관리가 어려울 때.
- **빠른 변화와 지속적인 배포가 필요할 때:** 시장 요구사항에 맞춰 특정 기능을 빠르게 업데이트해야 할 때.
- **대규모의 개발 조직:** 팀을 기능별로 분리하고 독립적인 개발을 진행해야 할 때.

# 헥사고날 아키텍처

헥사고날 아키텍처는 MSA나 모놀리식 같은 배포 방식이 아닌 코드의 구조와 설계 원칙에 관한 것이다. 애플리케이션의 핵심 비즈니스 로직을 외부 요소(UI, 데이터베이스, 외부 API 등)로부터 격리하여 의존성을 제어하는 것을 목표로 한다.

| **구분** | **장점 (Pros)** | **단점 (Cons)** |
| --- | --- | --- |
| **개발/구조** | 비즈니스 로직의 명확한 분리, 높은 유지보수성 | 초기 설계 및 설정에 추가적인 복잡성 (포트/어댑터 정의) |
| **테스트** | 외부 환경(DB, API) 없이 **순수 비즈니스 로직만** 단위 테스트 가능 (Mocking 용이) | 아키텍처 원칙을 준수하지 않으면 이점이 사라질 수 있음 |
| **기술 독립성** | 데이터베이스나 UI 기술을 쉽게 교체 가능함 | 아키텍처를 이해하고 적용하는 데 시간이 필요함 |
- **도메인 주도 설계 (DDD)와 함께:** 복잡한 비즈니스 로직을 명확하게 보호하고 관리해야 할 때.
- **기술 종속성을 최소화하고 싶을 때:** 데이터베이스나 외부 서비스가 자주 변경될 가능성이 있거나, 나중에 쉽게 교체할 수 있도록 설계하고 싶을 때.
- **높은 품질의 테스트 가능성이 중요할 때:** 비즈니스 규칙에 대한 빠르고 안정적인 단위 테스트가 필수적일 때.
